## Context

The repository currently runs a microservice stack through a single root `docker-compose.yml`, and several NestJS services use `TypeOrmModule.forRootAsync` with `synchronize: true` in their non-test PostgreSQL configuration. That means local startup can create or alter tables based on entity definitions instead of treating the database schema as an explicit prerequisite. The same compose file also wires the API gateway and frontend to depend on the entire backend stack, so developers pay the cost of booting unrelated services even when working on a narrow flow such as product browsing or cart behavior.

This change is cross-cutting because it touches eight backend services, root runtime orchestration, and developer-facing documentation. The design has to preserve the current architecture, ports, and full-stack path while making schema safety and on-demand startup explicit.

## Goals / Non-Goals

**Goals:**
- Prevent every backend service from mutating PostgreSQL schema during non-test startup.
- Keep test-only in-memory database synchronization where it is already used for isolated tests.
- Remove any implicit database bootstrap behavior such as synchronize-on-boot or migrations-on-boot from local runtime.
- Add explicit startup groups that let developers run product-only, cart, checkout, or full-stack workflows with short commands on Windows.
- Publish a feature-to-service dependency map that explains which services and infrastructure are needed for each workflow.

**Non-Goals:**
- Redesign service boundaries, API ownership, or the async workflow architecture.
- Delete or rebuild existing database schema.
- Introduce a new migration system or automatically reconcile schema drift.
- Replace Docker Compose with a different orchestration tool.

## Decisions

### Decision: Standardize non-test TypeORM runtime on schema-safe settings

All backend services that use TypeORM will follow the same non-test runtime policy:
- `synchronize: false`
- `migrationsRun: false`
- `dropSchema: false`

For services that currently have test-only SQL.js support, the existing test branch can keep `synchronize: true` so isolated test databases still build their in-memory schema. For services without that branch today, the change only needs to harden the live PostgreSQL configuration and must not introduce new schema mutation paths.

Rationale:
- This directly addresses the root cause: runtime DDL generated by ORM configuration.
- The policy is easy to audit because the safety guarantee lives in one obvious setting per service.
- It preserves current application wiring and avoids a larger refactor.

Alternatives considered:
- Environment-variable gating for synchronization in local dev. Rejected because it still leaves a runtime path that can silently mutate schema when misconfigured.
- Automatic migrations on boot. Rejected because it still changes the real database structure during service startup and violates the requirement for explicit schema management.

### Decision: Audit for non-TypeORM schema bootstrap and remove it rather than hiding it

The implementation should explicitly search for raw DDL, `queryRunner` schema helpers, and migration bootstrap hooks. If none exist, the change should document that only TypeORM synchronization was responsible. If any bootstrap code is found, it should be removed or disabled in non-test runtime rather than wrapped in a feature flag.

Rationale:
- The requirement is broader than one TypeORM flag; the runtime must have no silent schema patch path.
- An audit-driven approach makes the change verifiable and keeps the artifact grounded in actual code.

Alternatives considered:
- Rely only on a narrow search for `synchronize`. Rejected because it could miss manual startup DDL paths.

### Decision: Keep a single compose file and add profile-based startup groups plus short wrapper commands

The existing root `docker-compose.yml` remains the source of truth, but services will be assigned explicit profiles such as `core`, `product-flow`, `cart-flow`, `checkout-flow`, and `full-stack`. Root-level helper scripts or package scripts will expose short commands that call the corresponding `docker compose --profile ... up` workflow so developers do not need to memorize long invocations.

The startup groups should be cumulative and explicit:
- `core`: shared developer entrypoints needed for most frontend-backed work.
- `product-flow`: `core` plus product and inventory dependencies.
- `cart-flow`: `product-flow` plus cart support.
- `checkout-flow`: `cart-flow` plus user, order, payment, notification, and required messaging dependencies.
- `full-stack`: the current everything-on workflow.

Rationale:
- Compose profiles preserve the existing architecture and file layout.
- The root compose file still supports the current full-stack command.
- Wrapper commands keep the workflow friendly on Windows and avoid vague manual startup steps.

Alternatives considered:
- Multiple compose files. Rejected because it increases duplication and drift across service definitions.
- Documentation-only grouping. Rejected because the requirement asks for a simple workflow with obvious commands, not manual interpretation.

### Decision: Make feature dependency ownership explicit in documentation and verification

The change will add a concrete feature-to-service map that states, for each feature, the owning service, required running services, and whether Redis or RabbitMQ is needed. Validation steps will exercise the smallest startup group that should satisfy each workflow and confirm unrelated services stay stopped.

Rationale:
- The current stack wiring is not obvious from service names alone.
- Making dependencies explicit reduces accidental over-startup and gives developers a reliable troubleshooting reference.

Alternatives considered:
- Leave dependency discovery to README prose scattered across service docs. Rejected because it is incomplete for feature-scoped startup decisions.

## Risks / Trade-offs

- [Existing local databases may depend on auto-created tables] -> Mitigation: document that runtime now expects schema to pre-exist and call out any service whose local database must be prepared before startup.
- [Compose profile dependencies can accidentally over-start infrastructure or unrelated services] -> Mitigation: define groups from actual feature flows and verify each command against expected running containers.
- [API gateway or frontend assumptions may still point at services that are intentionally down] -> Mitigation: keep group commands aligned with supported workflows and document that unsupported routes will fail unless their owning profile is started.
- [Developers may bypass wrapper commands and use the wrong profile mix] -> Mitigation: keep `full-stack` available, provide short canonical commands, and document exact profile contents.

## Migration Plan

1. Audit all backend services for `synchronize`, `migrationsRun`, raw schema bootstrap code, and other startup DDL paths.
2. Update non-test TypeORM configuration in each affected service to disable schema mutation and explicit migration execution on boot.
3. Add compose profiles and short root commands for `core`, `product-flow`, `cart-flow`, `checkout-flow`, and `full-stack`.
4. Update root documentation with the startup group matrix, feature-to-service map, and schema safety guarantee.
5. Validate each startup mode and confirm full-stack startup still works.

Rollback:
- Revert the compose profile and script changes if a startup group is incomplete.
- Revert individual service config changes only if they break startup against an already-correct schema; do not reintroduce auto-sync as a permanent fix.

## Open Questions

- Whether the API gateway should live in `core` alone or only in workflow groups that include at least one downstream business service.
- Whether the frontend `my-app` should be included in every profile or remain an opt-in companion command for backend-only debugging.
- Whether any existing local seed/setup process needs to be documented separately once runtime schema mutation is removed.
