## 4. Seed Data Contract

### Auth Service

- Upsert admin auth user with `id=adminId`, role `admin`, active.
- Upsert seller auth user with `id=sellerId`, role `seller`, active.
- Upsert buyer auth user with `id=buyerId`, role `customer`, active.
- Do not write real passwords/tokens to docs.

### User Service

- Upsert buyer profile/address records if the runtime checkout requires address ownership.
- Upsert seller profile if seller/admin APIs require seller profile lookup.

### Store Service

- Upsert shop with `id=shopId`, `seller_id=sellerId`, status `approved`.

### Product Service / MongoDB Atlas

- Upsert category with `id=categoryId`.
- Upsert product with `id=productId`, `shopId=shopId`, `sellerId=sellerId`, `approvalStatus=approved`, `isActive=true`.
- Upsert image with `id=imageId`, `productId=productId`.
- Upsert variant with `id=variantId`, `productId=productId`, `isActive=true`.

### Inventory Service

- Upsert inventory item linked to `shopId`, `productId`, `variantId`, stock > 0, reserved_stock reset only for this seed item.
- Do not drop or alter unrelated inventory rows.

### Cart Service

- Prefer API Gateway add-cart endpoint for runtime verification.
- If needed for precondition setup, upsert `cart_state` for `buyerId` and `ownerKey=cart:user:{buyerId}` without duplicating.

### Order / Payment

- Do not pre-create final checkout order unless an API blocker prevents checkout.
- Checkout E2E should create `orders`, `shop_orders`, `shop_order_items`.
- Payment transaction should be created through runtime flow; direct insert only if service blocker is documented.

## 5. Execution Plan

1. Verify Docker services and Atlas connectivity.
2. Create/upsert unified seed data with a controlled script.
3. Start required runtime services through Docker Compose.
4. Execute E2E through API Gateway as much as possible.
5. Verify DB state after each critical step:
   - cart contains seeded variant
   - order rows exist and are linked
   - payment transaction exists or failure is documented
   - inventory reserved/stock state changes or failure is documented
6. Generate reports with fixed IDs and runtime IDs.

## 8. Initial Risks From Phase 5 Docs

- Existing Phase 5E only verified list APIs.
- Existing product MongoDB/Atlas collections were empty before Phase 5E.1.

- Checkout depends on consistent IDs across auth, store, product, inventory, cart, order, and payment databases.
- Some routes may require headers/JWT assumptions; test script must record the auth method used.
