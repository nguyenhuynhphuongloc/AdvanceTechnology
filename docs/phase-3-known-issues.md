# Phase 3 Known Issues

> **Ngày**: 16 May 2026
> **Phase**: 3 — Checkout, Order Split & Payment Foundation

---

## 1. Service-to-Service Auth (High Priority)

**Issue**: Internal API endpoints (`/api/v1/internal/*`) have no service-to-service authentication. Anyone with network access can call reserve/release/commit inventory.

**Impact**: Stock can be manipulated externally.

**Resolution**: Add internal API key header (e.g., `x-internal-api-key`) validated at each internal endpoint. Future phase.

---

## 2. No Idempotency on Checkout

**Issue**: If checkout API is called twice with the same cart, two orders may be created.

**Impact**: Duplicate orders if user double-clicks or retries.

**Resolution**: Add idempotency key header support. Phase 4+.

---

## 3. Seller/Admin Order UI Still Uses localStorage

**Issue**: `/seller/orders` and `/admin/orders` frontend pages still use localStorage mock.

**Impact**: Sellers and admins can't see real orders.

**Resolution**: Update seller and admin order pages to call real APIs. Phase 4.

---

## 4. No Payment Split Per Shop

**Issue**: Phase 3 creates a single payment record for the entire order. No per-shop payment split.

**Impact**: Platform can't calculate per-shop revenue.

**Resolution**: Phase 8 — Payment split & Settlement.

---

## 5. No Commission Calculation

**Issue**: Platform commission not calculated or deducted.

**Impact**: No seller payout logic.

**Resolution**: Phase 8 — Commission & Settlement.

---

## 6. Refund Flow Not Implemented

**Issue**: No refund request, approval, or processing.

**Impact**: If buyer cancels after payment, no refund mechanism exists.

**Resolution**: Phase 8 — Refund flow.

---

## 7. Legacy Order Migration

**Issue**: Old orders (with int userId) and new orders (with UUID buyerId) coexist. Admin list may show different data shapes.

**Impact**: Admin might see inconsistent data.

**Resolution**: Migration script to convert old orders, or filter old orders out of admin list. Future phase.

---

## 8. Checkout Shipping Address Not Persisted

**Issue**: Shipping address is snapshotted in Order but not stored as a separate Address entity.

**Impact**: No address management for repeat purchases.

**Resolution**: Phase 5+ — Address management for buyers.

---

## 9. Stock Validation Race Condition

**Issue**: Between checking stock availability and reserving it, another checkout could consume stock.

**Impact**: Potential overselling under high concurrency.

**Resolution**: Database-level locking on inventory items or optimistic locking. Future phase.

---

## 10. Cart Fetched by buyerId (x-user-id Header)

**Issue**: `fetchCart` uses `x-user-id` header, which is forwarded from gateway. No JWT verification at order-service.

**Impact**: Buyer could potentially manipulate the user ID header.

**Resolution**: Validate JWT token and extract userId from it. Currently relies on gateway JWT guard being enforced.

---

## 11. No Order Number Uniqueness Enforcement on Concurrent Checkout

**Issue**: `generateOrderNumber()` uses random suffix — theoretically could collide under high concurrency.

**Impact**: Very low probability duplicate order numbers.

**Resolution**: Add DB unique constraint (already exists) or use UUID prefix. Acceptable risk.

---

## 12. COD Payment Status Not Updated

**Issue**: COD payment stays at `pending`. No webhook/status update when cash is collected.

**Impact**: Payment status never transitions to `paid` for COD.

**Resolution**: Phase 5+ — Order status update when delivery confirmed.
